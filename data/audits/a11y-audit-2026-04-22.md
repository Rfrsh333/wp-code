## Top-10 Fixes

### 1. BLOCKER — Form labels niet gekoppeld aan inputs

**WCAG:** 1.3.1 Info and Relationships, 4.1.2 Name, Role, Value

**Component:** `InschrijfFormulier.tsx` (FormInput, FormSelect, FormTextarea)
**Ook:** `PersoneelAanvragenWizard.tsx`, `contact/page.tsx`

**Probleem:** De `<label>` elementen bevatten tekst maar de bijbehorende `<input>` zit in een sibling `<div>`. Er is geen `htmlFor`/`id` koppeling. Screen readers melden "edit text" zonder te weten welk veld het is.

**Impact:** Screen reader gebruikers kunnen de inschrijf- en aanvraagformulieren NIET invullen. Dit is een complete blokkade.

**Fix:**

```tsx
function FormInput({ label, id, required, error, ...props }) {
  return (
    <div>
      <label htmlFor={id} className="block text-sm font-medium mb-2">
        {label}
        {required && <span className="text-[#F27501]" aria-hidden="true"> *</span>}
        {required && <span className="sr-only"> (verplicht)</span>}
      </label>
      <div>
        <input
          id={id}
          aria-required={required || undefined}
          aria-invalid={!!error || undefined}
          aria-describedby={error ? `${id}-error` : undefined}
          {...props}
        />
      </div>
      {error && (
        <p id={`${id}-error`} className="text-red-500 text-sm mt-2" role="alert">
          {error}
        </p>
      )}
    </div>
  );
}
```

**Bestanden:**
- `src/components/InschrijfFormulier.tsx`
- `src/components/PersoneelAanvragenWizard.tsx`
- `src/app/contact/page.tsx`

### 2. BLOCKER — ToggleGroup buttons missen aria-pressed

**WCAG:** 4.1.2 Name, Role, Value

**Component:** `InschrijfFormulier.tsx` ToggleGroup

**Probleem:** Toggle buttons voor functies en talen hebben geen `aria-pressed` attribuut. Screen readers kunnen niet detecteren of een optie geselecteerd is.

**Impact:** Screen reader gebruikers kunnen niet zien welke functies/talen ze geselecteerd hebben.

**Fix:**

```tsx
<button
  type="button"
  onClick={() => toggle(option)}
  aria-pressed={isActive}
  className={isActive ? "bg-[#F27501] text-white ..." : "bg-white ..."}
>
  {option}
</button>
```

### 3. BLOCKER — Homepage knoppen zonder accessible names

**WCAG:** 4.1.2 Name, Role, Value

**Elementen:** Carousel/slider navigatieknoppen: `progressbar-prev`, `progressbar-next`, `testimonial-prev`, `testimonial-next`, `nav-prev-swiper`, `nav-next-swiper`

**Probleem:** Knoppen bevatten alleen een icoon/SVG zonder tekst of `aria-label`. Screen readers melden "button" zonder context.

**Impact:** Screen reader gebruikers weten niet wat deze knoppen doen.

**Fix:**

```tsx
<button className="progressbar-prev" aria-label="Vorige slide">
  <svg>...</svg>
</button>
<button className="progressbar-next" aria-label="Volgende slide">
  <svg>...</svg>
</button>
```

### 4. SERIOUS — Kleurcontrast failures op hele site

**WCAG:** 1.4.3 Contrast (Minimum)

**Omvang:** 67+ instances op homepage, doorlopend op contact, diensten, over-ons

**Hoofdoorzaken:**

| Huidig | Op achtergrond | Ratio | Probleem |
|--------|---------------|-------|----------|
| `text-neutral-400` (#a3a3a3) | wit (#fff) | 2.57:1 | Ver onder 4.5:1 minimum |
| `text-neutral-500` (#737373) | neutral-50 (#fafafa) | 4.39:1 | Net onder 4.5:1 |
| Teller-nummers | — | 1.15:1 | Extreem laag |
| `#F27501` (oranje) | wit (#fff) | ~3.2:1 | Faalt voor normaal tekst, OK voor groot tekst |

**Fix:**

```css
/* Vervang text-neutral-400 door text-neutral-600 */
/* #525252 op #ffffff = 7.45:1 — PASS */

/* Vervang text-neutral-500 door text-neutral-600 waar nodig */
/* #525252 op #fafafa = 7.21:1 — PASS */

/* Voor oranje merktekst op wit: gebruik #c45e00 */
/* #c45e00 op #ffffff ≈ 4.6:1 — PASS */
```

### 5. SERIOUS — Focus indicators te zwak

**WCAG:** 2.4.7 Focus Visible, 2.4.11 Focus Appearance (AAA maar best practice)

**Probleem:** Alle formulier-inputs gebruiken `focus:outline-none` wat de standaard browser focus indicator verwijdert. De vervangende `ring-4 ring-[#F27501]/10` (10% opacity) is nauwelijks zichtbaar.

**Impact:** Keyboard-gebruikers kunnen niet zien welk element focus heeft.

**Fix:**

```css
/* Vervang */
focus:outline-none focus:ring-4 focus:ring-[#F27501]/10

/* Door */
focus:outline-2 focus:outline-offset-2 focus:outline-[#F27501]
/* Of: */
focus:ring-2 focus:ring-[#F27501]/50
```

### 6. SERIOUS — Diensten dropdown niet keyboard-toegankelijk

**WCAG:** 2.1.1 Keyboard

**Component:** `Header.tsx` desktop navigatie

**Probleem:** Het dropdown menu voor "Diensten" gebruikt alleen CSS `group-hover`. Er zijn geen keyboard event handlers (Enter, Space, Arrow keys), geen `aria-expanded`, geen `aria-haspopup`. Keyboard-gebruikers kunnen de sub-items niet bereiken.

**Fix:**

```tsx
function DienstenDropdown() {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <div className="relative">
      <button
        aria-expanded={isOpen}
        aria-haspopup="true"
        onClick={() => setIsOpen(!isOpen)}
        onKeyDown={(e) => {
          if (e.key === 'Escape') setIsOpen(false);
          if (e.key === 'ArrowDown') {
            e.preventDefault();
            setIsOpen(true);
            // Focus first item
          }
        }}
      >
        Diensten
      </button>
      {isOpen && (
        <ul role="menu">
          <li role="menuitem"><Link href="/diensten/horeca">Horeca</Link></li>
          {/* ... */}
        </ul>
      )}
    </div>
  );
}
```

### 7. SERIOUS — Foutmeldingen niet aangekondigd

**WCAG:** 3.3.1 Error Identification, 4.1.3 Status Messages

**Probleem:** Foutmeldingen in formulieren verschijnen visueel maar worden niet aangekondigd door screen readers. Er ontbreekt:
- `role="alert"` of `aria-live="polite"` op foutmeldingen
- `aria-invalid="true"` op ongeldige invoervelden
- `aria-describedby` koppeling tussen veld en foutmelding

**Impact:** Screen reader gebruikers weten niet dat er een fout is opgetreden of welk veld gecorrigeerd moet worden.

**Fix:** Zie codevoorbeeld bij Fix #1 — de `role="alert"`, `aria-invalid`, en `aria-describedby` attributes zijn daar meegenomen.

### 8. SERIOUS — Geen focus management bij stap-wisseling

**WCAG:** 3.2.2 On Input

**Component:** Multi-step wizards (InschrijfFormulier, PersoneelAanvragenWizard, KostenCalculator)

**Probleem:** Wanneer een gebruiker op "Volgende" klikt in een multi-step formulier, blijft de focus op de oude positie. De nieuwe stap-content verschijnt buiten het scherm van de screen reader.

**Impact:** Screen reader gebruikers weten niet dat er nieuwe content geladen is en moeten handmatig terug navigeren.

**Fix:**

```tsx
const nextStep = () => {
  if (validateStep(currentStep)) {
    setCurrentStep((prev) => Math.min(prev + 1, totalSteps - 1));
    // Focus de heading van de nieuwe stap na render
    requestAnimationFrame(() => {
      const heading = document.querySelector('[data-step-heading]');
      if (heading instanceof HTMLElement) {
        heading.setAttribute('tabindex', '-1');
        heading.focus();
      }
    });
  }
};
```

### 9. MODERATE — Homepage mist h1

**WCAG:** 1.3.1 Info and Relationships

**Probleem:** De homepage heeft geen `<h1>` element. Axe meldt `page-has-heading-one`. De hero-titel wordt waarschijnlijk geladen via een dynamisch component dat geen `<h1>` rendert.

**Impact:** Screen reader gebruikers missen de hoofdtitel van de pagina. De heading-structuur is verstoord (h2 naar h6 overspringing).

**Fix:** Zorg dat het hero-component een `<h1>` rendert:

```tsx
<section className="hero">
  <h1 className="text-4xl md:text-6xl font-bold">
    {heroTitle}
  </h1>
</section>
```

### 10. MODERATE — Cookie consent banner niet WCAG-compliant

**WCAG:** 2.4.3 Focus Order, 4.1.2 Name, Role, Value

**Component:** `CookieConsent.tsx`

**Probleem:**
- Geen `role="dialog"` op de banner
- Geen `aria-label` om het doel te beschrijven
- Focus wordt niet naar de banner verplaatst bij verschijnen
- Geen focus trap — gebruikers kunnen voorbij de banner tabben

**Impact:** Keyboard- en screen reader gebruikers merken de cookie banner mogelijk niet op en geven onbewust geen toestemming.

**Fix:**

```tsx
<div
  role="dialog"
  aria-label="Cookie voorkeuren"
  aria-modal="true"
  ref={bannerRef}
>
  {/* Focus naar banner bij mount */}
  {/* Focus trap implementeren */}
  <p>Wij gebruiken cookies...</p>
  <button onClick={accept}>Accepteren</button>
  <button onClick={reject}>Weigeren</button>
</div>
```

```tsx
useEffect(() => {
  if (showBanner && bannerRef.current) {
    bannerRef.current.focus();
  }
}, [showBanner]);
```

## EAA Advies

### Valt TopTalent onder de European Accessibility Act?

**Waarschijnlijk ja.** TopTalent biedt online diensten aan waarbij overeenkomsten worden gesloten (uitzendovereenkomsten, bemiddeling). Dit valt onder "e-commerce diensten" in de zin van de EAA.

### Micro-onderneming uitzondering

De EAA bevat een uitzondering voor micro-ondernemingen (< 10 werknemers EN < EUR 2M jaaromzet). TopTalent is een uitzendbureau — het interne team is mogelijk klein, maar uitzendkrachten kunnen meetellen afhankelijk van interpretatie.

**ACTIE VEREIST:** Verifieer met KvK of TopTalent als micro-onderneming kwalificeert.

### Advies ongeacht EAA-status

De gevonden BLOCKER en SERIOUS issues moeten hoe dan ook gefixed worden:

1. **Formulieren zijn onbruikbaar** voor screen reader gebruikers — kandidaten met een visuele beperking kunnen zich niet inschrijven
2. **Discriminatierisico:** De Wet gelijke behandeling op grond van handicap of chronische ziekte (Wgbh/cz) geldt ongeacht EAA-status
3. **Best practice:** WCAG 2.1 AA is industriestandaard en wordt steeds vaker vereist door opdrachtgevers
